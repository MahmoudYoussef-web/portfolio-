Case Study

LogiBridge — B2B Delivery Integration Platform

Problem

Coordinating between shipping companies and last-mile delivery providers is traditionally a manual process — phone calls, spreadsheets, no shared source of truth. LogiBridge replaces that with a single API-driven platform, but the real engineering problem isn't the CRUD surface — it's guaranteeing correctness when multiple people (a company dispatcher, a delivery agent, a mobile app retrying a failed request) try to mutate the same order at the same time.

Architecture

Controller → Service → Repository → Entity

The system follows a layered architecture with domain logic living inside the entity layer (a lightweight DDD approach): services orchestrate business operations, but entities themselves enforce valid state transitions. An idempotency layer and a concurrency-control layer sit between the controller and the service layer, protecting the domain from unsafe duplicate or concurrent mutations before they ever reach business logic.

Key Design Decisions

Dual-layer idempotency.

Network retries are a fact of production systems — if a delivery agent's mobile app times out and retries an "accept order" request, the system must not process it twice. LogiBridge solves this two ways: a DB-backed layer (IdempotencyService) that stores the first response against a unique (key, userId) constraint and replays it on retry, and a Redis-backed layer for sub-millisecond deduplication using SET key IN_PROGRESS NX EX 300 as an atomic acquire. If two requests race on the same key at the database layer, the resulting constraint violation is caught and the losing thread simply returns the winner's stored response instead of erroring.

Two independent concurrency mechanisms, not one.

Pessimistic locking (PESSIMISTIC_WRITE) is acquired on the order row before any state-changing operation — accept, reject, cancel, status update — so concurrent requests wait rather than race. Independently, the Order entity also carries an optimistic @Version field, so if a concurrent write does slip through, the second transaction fails fast with a retryable exception instead of silently overwriting data. The state machine (OrderStatus.canTransitionTo()) acts as a third, domain-level guard — even if both locking mechanisms were somehow bypassed, an invalid transition is rejected before persistence.

Roles are never client-supplied.

The role a user receives is determined entirely by which registration endpoint they call (/register/company vs /register/delivery), not by a field in the request body — closing off a privilege-escalation vector at the design level rather than relying on validation to catch it.

Refresh token rotation with reuse detection.

Every token refresh invalidates the previous token and issues a new one. If a previously-revoked token is ever presented again, that's treated as a signal of token theft, and all sessions for that user are invalidated — not just the one session.

Challenges & Tradeoffs

Choosing pessimistic AND optimistic locking, not one or the other.

Pessimistic locking alone would be sufficient in theory, but relying on a single mechanism means a single class of bug (e.g., a code path that forgets to acquire the lock) has no safety net. Layering optimistic versioning on top costs a small amount of complexity but means a missed lock fails loudly (an exception) instead of silently corrupting order state.

Immutable audit trail.

OrderTracking records are made genuinely immutable — a @PreUpdate hook throws UnsupportedOperationException if any update is attempted — enforced at the ORM level, not just by convention. The tradeoff is that correcting a bad tracking record requires inserting a compensating record rather than editing history, which is the correct behavior for an audit trail but does add a small amount of friction during debugging.

Results

  • Order accept/reject/cancel operations are provably exactly-once under concurrent access and network retries
  • Zero privilege-escalation surface for role assignment (role is never accepted from client input)
  • Tamper-proof audit trail enforced at the ORM layer, not just application logic
  • Test suite covers the three highest-risk areas specifically: auth logic, idempotency guarantees, and state-machine correctness — including a full end-to-end integration test that creates an order, accepts it, then retries the accept with the same idempotency key and asserts identical responses
