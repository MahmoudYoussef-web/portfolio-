Case Study

E-Commerce Backend System

Problem

An e-commerce backend touches almost every hard problem in backend engineering at once: payments that must never double-charge or double-fulfill, inventory that must never oversell, and financial records that must reconcile exactly. This project was built to handle the full commerce lifecycle — auth, catalog, cart, orders, Stripe payments, inventory, and financial reporting — with production-level reliability guarantees, not just a happy-path demo.

Architecture

REST Controllers → Service Layer → Repository (JPA/MySQL)
                                 ↘ Event Layer (OrderCreated, PaymentCompleted)
                                     → Listeners/Handlers (inventory sync, finance journal entries)

Design principles applied throughout: DTOs at the API boundary (entities never leave the service layer), centralized exception handling via @RestControllerAdvice, cache logic isolated into a dedicated ProductCacheService, and interface-first services for testability.

Key Design Decisions

Idempotent Stripe webhook handling.

Payment providers can and do deliver the same webhook more than once. The webhook endpoint verifies the Stripe-Signature header, independently validates the amount, currency, and ownership server-side before confirming an order, and safely ignores duplicate webhook deliveries — so a network hiccup on Stripe's side can never result in double-fulfilling an order.

Two-phase inventory reservation instead of pessimistic row locks.

Rather than locking product rows during checkout (which would hurt throughput under load), inventory uses an event-driven two-phase approach: creating an order publishes an OrderCreatedEvent, which triggers a temporary StockReservation that holds stock while payment is pending. A scheduled job releases reservations for orders that never complete payment. Once PaymentCompletedEvent fires, the reservation converts into a permanent StockMovement record. This prevents overselling without requiring locks on the hot path.

Address snapshotting for order integrity.

Shipping addresses are frozen at the moment of purchase rather than referenced live from the user's address book. This is a small decision with an outsized correctness benefit: a customer editing or deleting their saved address after ordering can never retroactively corrupt a historical order record.

Double-entry bookkeeping for financial reporting.

Revenue isn't just a summed column — it's modeled with ChartOfAccount and JournalEntry entities following double-entry accounting principles, which makes the financial reports auditable and internally consistent rather than just "close enough."

Challenges & Tradeoffs

Cache invalidation granularity.

Product reads are cached in Redis through a dedicated ProductCacheService, with individual product entries kept warm via @CachePut on writes, while paginated listing caches are invalidated entirely (@CacheEvict(allEntries = true)) on any product write. This is a deliberate tradeoff: precise invalidation of every possible page/filter combination would be complex and error-prone, so the system accepts a slightly more aggressive cache eviction on the paginated endpoints in exchange for guaranteed correctness.

Two webhook security schemes, not one.

Stripe events are verified via Stripe's own signature header, while internal webhooks use a separate HMAC-based scheme with timestamp validation — both funneled through a shared WebhookValidator abstraction. Maintaining two schemes adds a small amount of code, but treating internal and third-party webhook trust boundaries identically would have been the wrong tradeoff.

Results

  • Payment webhooks are provably idempotent — duplicate Stripe deliveries never double-fulfill an order
  • Zero overselling under concurrent checkout load, without pessimistic locks on the product table
  • Financial reports are internally consistent by construction (double-entry bookkeeping), not just summed estimates
  • Historical orders remain correct even if a customer's address book changes after purchase
