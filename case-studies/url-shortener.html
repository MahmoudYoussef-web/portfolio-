Case Study

Distributed URL Shortener

Problem

Most URL shorteners look simple on the surface, but at scale they hit real distributed-systems problems: how do you generate unique short codes without a single point of contention, how do you serve redirects fast enough that the cache layer doesn't become the bottleneck, and how do you rate-limit abusive traffic without race conditions. This project was built specifically to confront those problems rather than ship a toy CRUD app — the goal was to simulate how a system like Bitly operates under real traffic, not just replicate its feature list.

Architecture

Client
   ↓
Spring Boot (Stateless)
   ↓
Redis
  ├── Cache (read acceleration)
  ├── ID Generator (atomic INCR)
  └── Rate Limiter (Lua script)
   ↓
Shard Router (hash % N)
   ↓
MySQL Shards (source of truth)

The service layer is stateless, which means any instance can handle any request — all coordination happens through Redis and the shard router, not through in-memory state.

Key Design Decisions

No ORM — Raw JDBC.

JPA was deliberately excluded. The repository layer uses JdbcTemplate directly with explicit shard routing. This trades away Hibernate's convenience for full control over query execution, eliminates N+1 risk by construction, and keeps the persistence layer transparent — every query is visible, not generated.

Distributed ID generation via Redis INCR.

Rather than relying on database auto-increment (which doesn't work cleanly across shards) or UUIDs (which are long and not URL-friendly), the system uses Redis's atomic INCR command to generate a globally unique integer, then encodes it to Base62 for a short, URL-safe code. If Redis becomes unavailable, an AtomicLong fallback keeps the system running in degraded mode instead of failing outright.

Cache-aside, not cache-through.

Redis is treated strictly as a performance layer — MySQL is always the source of truth. On a redirect, a cache hit returns immediately; a cache miss queries the database, populates the cache, and returns. Critically, the cache TTL is derived from the URL's actual expires_at value in the database rather than a fixed offset, so the cache and the database can never disagree about whether a link is still valid.

Deterministic sharding with no routing table.

Shard selection is computed as Math.abs(shortCode.hashCode()) % SHARD_COUNT. Because the formula is deterministic, any node can independently route any request without consulting a shared routing table — removing a potential single point of failure. Per-shard JdbcTemplate instances are created lazily and cached, so adding a shard is a configuration change, not a code change.

Challenges & Tradeoffs

The rate limiter's race condition.

The naive approach to rate limiting — INCR the counter, then EXPIRE it — has a subtle bug: if the process crashes between the two commands, the key never expires and the rate limit becomes permanent for that client. The fix was to move both operations into a single atomic Lua script executed by Redis, so the increment and the expiry either both happen or neither does. This is a good example of a bug that only shows up under real concurrent load, not in a quick manual test.

Fail-open vs fail-closed.

For rate limiting specifically, the system is fail-open: if Redis is unavailable, requests pass through rather than being blocked. This was a deliberate tradeoff — an outage in Redis should degrade the user experience (no rate limiting), not take down the whole API.

Expiration correctness under cache staleness.

Since Redis TTL is only a hint and MySQL is authoritative, every database query explicitly filters WHERE expires_at IS NULL OR expires_at > NOW() at the SQL level. This guarantees an expired link is never served, even in the edge case of a Redis cache miss right after expiry.

Results

  • Zero ID collisions by construction (atomic Redis INCR, not a probabilistic scheme)
  • No N+1 query risk, by using raw JDBC instead of an ORM
  • Race-condition-free rate limiting via a single atomic Lua script
  • Horizontal scaling requires only a configuration change (incrementing SHARD_COUNT and adding a datasource entry) — no code changes to the routing logic
